## 1. บทนำ (Introduction)

### 1.1 วัตถุประสงค์ (Purpose)
กำหนดความต้องการเชิงฟังก์ชันและไม่ใช่ฟังก์ชันของระบบ **ALPR** ซึ่งให้ครูผู้สอน (CAT) อัปโหลดแผนการจัดการเรียนรู้ แล้วให้ AI วิเคราะห์เบื้องต้นตามเกณฑ์ AIPACK 5 องค์ประกอบ พร้อมเสนอคะแนนและร่องรอยหลักฐาน จากนั้นครูพี่เลี้ยง (CAM) ตรวจสอบ ยืนยัน หรือปรับแก้ ก่อนออกผลประเมินและ PLC Action Trigger

### 1.2 ขอบเขตของระบบ (Scope)
**อยู่ในขอบเขต (In-scope):**
- อัปโหลดไฟล์แผนการสอน (PDF/DOCX)
- สกัดข้อความ + ตรวจเช็กลิสต์หลักฐาน 5 รายการอัตโนมัติ
- AI ประเมิน 5 องค์ประกอบ (แต่ละองค์ประกอบ 1–4 คะแนน) พร้อม **อ้างอิงข้อความ/หน้า** เป็นหลักฐาน
- หน้าให้ CAM ยืนยัน/แก้คะแนน + เขียน Feedback เชิงกัลยาณมิตร
- คำนวณคะแนนรวม /20 + แปลผล 4 ระดับ + PLC Action Trigger อัตโนมัติ
- ออกรายงานผลประเมิน (PDF) + ประวัติการประเมิน

**นอกขอบเขต (Out-of-scope) รอบนี้:**
- ฟอร์มฉบับที่ 1 (สังเกตการสอน) และฉบับที่ 2 (ประเมิน CAM)
- บทบาทนักวิจัย/ผู้บริหาร, dashboard เชิงสถิติข้ามโรงเรียน
- การประเมินอัตโนมัติเต็มรูปแบบโดยไม่มีมนุษย์ยืนยัน

### 1.3 คำนิยามและตัวย่อ (Definitions)
| คำ | ความหมาย |
|----|----------|
| CAT | Coachee — ครูผู้สอน/ผู้จัดทำแผน (อัปโหลดแผน) |
| CAM | Coach/Mentor — ครูพี่เลี้ยง (ผู้ตรวจประเมินและยืนยันผล) |
| AIPACK | กรอบบูรณาการ AI + PACK (CK เนื้อหา, AIK ความรู้ AI) |
| RL / CT | Reading Literacy / Critical Thinking |
| Evidence Trail | ร่องรอยหลักฐาน (ข้อความ+เลขหน้าจากแผน) ที่ยืนยันคะแนน |
| PLC Action Trigger | แนวทางขับเคลื่อนที่ผูกกับแถบคะแนนโดยอัตโนมัติ |
| Analytic Rubric | เกณฑ์ให้คะแนนแบบแยกองค์ประกอบ |

### 1.4 หลักการออกแบบที่ต้องคงไว้ (Design Principles จากต้นฉบับ)
1. **Evidence-based เสมอ** — ทุกคะแนนต้องมี Evidence Trail อ้างอิงจากตัวแผน (AI ต้องยกข้อความ/หน้า ห้ามให้คะแนนลอย)
2. **Human-in-the-loop** — คะแนนสุดท้ายเป็นของ CAM เท่านั้น AI เป็นเพียงผู้ช่วยเสนอ
3. **เกณฑ์ 4 ระดับ** — ต้นแบบสร้างสรรค์(4) / เชี่ยวชาญช่ำชอง(3) / บ่มเพาะทักษะ(2) / เริ่มจุดประกาย(1)
4. **Feedback เชิงกัลยาณมิตร** — ผลลัพธ์ต้องมีทั้งจุดเด่น (Strengths) และจุดเติมเต็ม (Areas for Growth)

## 2. ภาพรวมระบบ (Overall Description)

### 2.1 มุมมองผลิตภัณฑ์ (Product Perspective)
เว็บแอปพลิเคชันแบบ multi-user; CAT อัปโหลดและติดตามผล, CAM ตรวจและยืนยัน. องค์ประกอบหลัก:
```
[CAT] ──upload──► [Ingestion/สกัดข้อความ] ──► [AI Evaluator (LLM)]
                                                      │ เสนอคะแนน+Evidence
                                                      ▼
                          [CAM Review UI] ──ยืนยัน/แก้──► [Scoring & PLC Engine]
                                                      │
                                                      ▼
                                   [รายงานผล /20 + Feedback] ──► [CAT]
```

### 2.2 กลุ่มผู้ใช้และคุณลักษณะ (User Classes)
| บทบาท | สิทธิ์หลัก |
|-------|-----------|
| **CAT** | อัปโหลดแผน, ดูสถานะ, ดูผลประเมินและ Feedback, แก้ไข/อัปโหลดฉบับปรับปรุง |
| **CAM** | ดูรายการแผนที่รอตรวจ, เรียกผล AI, ยืนยัน/ปรับคะแนนรายองค์ประกอบ, เขียน Feedback, ปิดผลและลงนามดิจิทัล |

### 2.3 ข้อสมมติและข้อจำกัด (Assumptions & Constraints)
- แผนอัปโหลดเป็นภาษาไทย รูปแบบ PDF (รวมสแกน) หรือ DOCX ขนาด ≤ 20 MB
- ต้องรองรับ **OCR** สำหรับ PDF สแกน (แผนบางเล่มเป็นภาพ)
- ผล AI ถือเป็น "ข้อเสนอ" (draft) เท่านั้น — ผลจะ valid ต่อเมื่อ CAM ยืนยัน
- ต้องเก็บ audit log การเปลี่ยนคะแนน (AI เสนอ → CAM แก้เป็นเท่าไร) เพื่อการวิจัย
- **การ deploy:** ระบบไม่ได้อยู่ที่ root ของโดเมน แต่เป็น **sub-folder/sub-path ใต้โดเมนที่มีอยู่แล้ว** (เช่น `https://<domain>/<basePath>/` — ค่า `basePath` ต้องกำหนดผ่าน environment variable ไม่ hardcode) ดูข้อกำหนดใน NFR-9
- **สภาพแวดล้อมจริงที่ยืนยันแล้ว:** อยู่ร่วมโดเมนกับ **DigiNest LMS** (`craftai.ksu.ac.th`) ซึ่งเป็น Next.js รันใต้ basePath `/lms` (asset `/lms/_next/...`) — ALPR ควรเป็นโมดูลรูปแบบเดียวกัน (เช่น `/aipack` หรือ sub-route ใต้ `/lms`) และใช้ธีม/ฟอนต์เดียวกัน (`Prompt` + `IBM Plex Sans Thai`, hue 225) ดู [UX/UI Design](UX_UI_Design_ALPR.md)

### 2.4 บริบทการ deploy แบบ Base Path (Sub-folder Hosting)
แอปถูกเสิร์ฟภายใต้ path ย่อยของโดเมนกลาง (reverse proxy/subdirectory) มิใช่โดเมน/ซับโดเมนอิสระ ทำให้ทุกส่วนที่อ้าง URL ต้องคำนึงถึง `basePath`:
```
https://<domain>/<basePath>/            → หน้าแอป
https://<domain>/<basePath>/api/...      → API/Server routes
https://<domain>/<basePath>/_next/...    → static assets (JS/CSS/ฟอนต์ไทย)
https://<domain>/<basePath>/reports/...  → ลิงก์ดาวน์โหลดรายงาน PDF
```

## 3. เกณฑ์การประเมิน (Evaluation Rubric — Data Model หลัก)

ระบบต้องเข้ารหัสเกณฑ์ 5 องค์ประกอบ องค์ประกอบละ 4 ระดับ (รวม 20 คะแนน) โดยแต่ละระดับผูกกับ **สัญญาณหลักฐานที่ตรวจจับได้**:

### C1 ความสอดคล้องกับมาตรฐาน (Alignment) — 4 คะแนน
- **4:** เชื่อมโยง CK(ตัวชี้วัด) ↔ RL/CT ↔ AIK(การใช้ AI) ชัดเจนกลมกลืน
- **3:** ระบุ CK + RL/CT แต่เชื่อมกับ AI ไม่ชัด
- **2:** ระบุ CK ปกติ กล่าวถึง RL/CT/AI ผิวเผิน
- **1:** มีแต่ CK ดั้งเดิม ไม่กล่าวถึง RL/CT/เทคโนโลยี

### C2 จุดประสงค์การเรียนรู้ (Objectives) — 4 คะแนน
- **4:** K-P-A ครบ, พฤติกรรม SMART วัดได้, เจาะจง RL/CT, เจตคติ Growth Mindset/จริยธรรม AI
- **3:** K-P-A ครบ ระบุทักษะคิด/อ่าน แต่ใช้คำกริยาวัดยาก ("เข้าใจ", "ตระหนัก")
- **2:** เน้น K + P พื้นฐาน ไม่เจาะจง RL/CT
- **1:** กว้างเกินไป วัดผลไม่ได้

### C3 กระบวนการจัดการเรียนรู้ (Process) — 4 คะแนน
- **4:** เขียนชุดคำถาม Socratic ในขั้นตอนชัด + กิจกรรมวิเคราะห์หลักฐาน + ระบุวิธีชม Growth Mindset + PERMA
- **3:** มี Active Learning + คำถามวิเคราะห์ แต่ไม่ระบุชุดคำถาม/ขาดวิธีสร้างบรรยากาศบวก
- **2:** เน้นครูบรรยาย ผู้เรียนตอบสั้น/ทำใบงานท้ายชั่วโมง
- **1:** ไม่มีขั้นตอนกิจกรรมชัด ระบุแค่หัวข้อ/ลอกตำรา

### C4 การใช้สื่อและเทคโนโลยี (Media & AI Tools) — 4 คะแนน
- **4:** แนบ Prompt จริง + AI เป็น Devil's Advocate + ขั้นตอนให้ผู้เรียน Fact-check
- **3:** ใช้สื่อจาก AI/แพลตฟอร์ม แต่ไม่มีขั้นตอนตรวจสอบความน่าเชื่อถือ
- **2:** ใช้ AI แค่ช่วยครูทำสไลด์/ความสวยงาม ผู้เรียนไม่ได้ใช้
- **1:** ไม่ระบุการใช้สื่อ/ไม่สอดคล้องเป้าหมาย

### C5 การประเมินผล (Evaluation & Assessment) — 4 คะแนน
- **4:** มี Formative+Summative + แนบ Analytic Rubric วัด RL/CT + ชิ้นงานเก็บร่องรอยการคิด
- **3:** ระบุวิธีวัด RL/CT + มีเครื่องมือ แต่เป็น Holistic Rubric กว้าง
- **2:** วัดแค่ความรู้ความจำ (ปรนัย) ไม่วัด RL/CT
- **1:** ไม่ระบุวิธี/เครื่องมือ หรือไม่สอดคล้องจุดประสงค์

### เช็กลิสต์หลักฐาน 5 รายการ (Pre-flight Evidence Signals)
ระบบตรวจอัตโนมัติ (พบ/ไม่พบ) และแสดงให้ CAM เห็นก่อนให้คะแนน:
`[1]` ระบุเครื่องมือ AI/แพลตฟอร์ม · `[2]` แนบ Prompt · `[3]` ชุดคำถาม Socratic · `[4]` Analytic Rubric · `[5]` ใบงานร่องรอยการคิด

### การแปลผลและ PLC Action Trigger
| ช่วงคะแนน | ระดับ | PLC Action (อัตโนมัติ) |
|-----------|-------|------------------------|
| 17–20 | ต้นแบบสร้างสรรค์ | ยกเป็น Best Practice ประจำสถานศึกษา / วิทยากรแกนนำ |
| 13–16 | เชี่ยวชาญช่ำชอง | ปรับถ้อยคำ Socratic + เกณฑ์วัดให้คมชัด → ใช้สอนได้ทันที |
| 9–12 | บ่มเพาะทักษะ | **⚠️ ห้ามนำแผนไปใช้สอนทันที** — นัด Mentor-Mentee เติมเครื่องมือ |
| 5–8 | เริ่มจุดประกาย | ดึงเข้าวง PLC ใหญ่ Co-design ปรับโครงสร้างแผนใหม่ทั้งหมด |

## 4. ความต้องการเชิงฟังก์ชัน (Functional Requirements)

### FR-1 การจัดการบัญชีและบทบาท (Authentication & Roles)
- **FR-1.1 เข้าสู่ระบบด้วย Google (Gmail) — วิธีหลัก:** ระบบต้องรองรับ **Sign in with Google (OAuth 2.0 / OpenID Connect)** เป็นช่องทางหลัก ผู้ใช้ล็อกอินด้วยบัญชี Gmail/Google Workspace โดยไม่ต้องตั้งรหัสผ่านแยก
  - FR-1.1.1 ปุ่ม "เข้าสู่ระบบด้วย Google" บนหน้า login → เปลี่ยนเส้นทางไปหน้ายินยอมของ Google → กลับมาที่ระบบ (callback) พร้อม token
  - FR-1.1.2 ดึงข้อมูลจากโปรไฟล์ Google: อีเมล, ชื่อ, รูปโปรไฟล์ (ใช้แสดงใน UI)
  - FR-1.1.3 **จำกัดโดเมน (Domain allow-list):** อนุญาตเฉพาะอีเมลในโดเมนที่กำหนด (เช่น `@ksu.ac.th` หรือรายการที่ตั้งค่าไว้) — อีเมลนอกรายการถูกปฏิเสธพร้อมข้อความชัดเจน
  - FR-1.1.4 ผู้ใช้ครั้งแรก (อีเมลผ่าน allow-list แต่ยังไม่มีบัญชีในระบบ) → สร้างบัญชีอัตโนมัติ (JIT provisioning) และรอ admin/CAM กำหนดบทบาท หรือกำหนดบทบาทเริ่มต้นตามนโยบาย
- **FR-1.2 บทบาท:** ระบบรองรับ 2 บทบาท (CAT, CAM); **บทบาทผูกกับบัญชีในระบบ ไม่ได้มาจาก Google** (Google ยืนยันตัวตนเท่านั้น ระบบเป็นผู้กำหนดสิทธิ์)
- FR-1.3 CAM ผูกกับ CAT ที่ตนดูแลได้ (คู่ Mentor-Mentee)
- FR-1.4 ผู้ใช้เห็นเฉพาะข้อมูลของตน/ผู้ที่ตนดูแล
- **FR-1.5 Session & ออกจากระบบ:** สร้าง session หลัง OAuth สำเร็จ, cookie ตั้ง `Path=<basePath>` (ดู NFR-9.5), มีปุ่มออกจากระบบที่เพิกถอน session ฝั่งเซิร์ฟเวอร์
- **FR-1.6 (ทางเลือก/อนาคต):** อาจรองรับล็อกอินด้วยอีเมล+รหัสผ่านสำรอง หรือ SSO เดียวกับ DigiNest LMS ในภายหลัง — รอบนี้ **Google เป็นวิธีเดียว**

### FR-2 การอัปโหลดแผน (CAT)
- FR-2.1 CAT อัปโหลดไฟล์ PDF/DOCX (≤20 MB) พร้อมกรอกข้อมูลส่วนที่ 1 (ชื่อ, รายวิชา/เรื่อง, วันที่)
- FR-2.2 ระบบตรวจชนิด/ขนาดไฟล์ และแจ้งข้อผิดพลาดที่ชัดเจน
- FR-2.3 รองรับหลายเวอร์ชันของแผนเดียวกัน (v1, v2 หลังปรับปรุง) และเก็บประวัติ
- FR-2.4 CAT ดูสถานะได้: `อัปโหลดแล้ว → AI กำลังวิเคราะห์ → รอ CAM ตรวจ → ประเมินเสร็จ`

### FR-3 การสกัดข้อมูลและตรวจเช็กลิสต์ (System)
- FR-3.1 สกัดข้อความจากไฟล์ (มี OCR fallback สำหรับ PDF สแกน)
- FR-3.2 ตรวจเช็กลิสต์หลักฐาน 5 รายการ พร้อมชี้ตำแหน่ง/หน้าที่พบ (พบ/ไม่พบ/ไม่แน่ใจ)

### FR-4 การประเมินด้วย AI (AI Evaluator)
- FR-4.1 สำหรับแต่ละองค์ประกอบ C1–C5 AI ต้องเสนอ: **ระดับคะแนน (1–4)** + **เหตุผลสั้น** + **Evidence Trail (ข้อความอ้างอิง + เลขหน้า)**
- FR-4.2 หากไม่พบหลักฐานเพียงพอ AI ต้องตอบ "ไม่พบหลักฐาน" และเสนอระดับต่ำ ห้าม "เดา" ให้คะแนนสูง
- FR-4.3 AI แสดง **ระดับความเชื่อมั่น (confidence)** ต่อองค์ประกอบ เพื่อให้ CAM โฟกัสจุดที่ต้องพิจารณา
- FR-4.4 ผลจาก AI ถูกทำเครื่องหมายเป็น *ข้อเสนอ (draft)* จนกว่า CAM จะยืนยัน
- FR-4.5 ผลลัพธ์ AI ต้องเป็น structured output (JSON schema ตายตัว) เพื่อ map เข้าฟอร์มโดยตรง

### FR-5 การตรวจสอบและยืนยันโดย CAM (Human-in-the-loop)
- FR-5.1 CAM เห็นคะแนนที่ AI เสนอเทียบกับ Evidence Trail แบบ side-by-side กับตัวแผน
- FR-5.2 CAM **ยืนยัน** หรือ **ปรับ** คะแนนแต่ละองค์ประกอบได้ พร้อมช่องบันทึกร่องรอยหลักฐานเพิ่ม
- FR-5.3 ระบบบันทึก audit ทุกครั้งที่คะแนนถูกแก้ (AI เสนอ X → CAM แก้เป็น Y) เพื่อการวิจัย
- FR-5.4 CAM เขียนช่องสรุป: 4.1 จุดเด่น (Strengths), 4.2 จุดตกลงร่วมเติมเต็ม (Areas for Growth)
- FR-5.5 CAM ลงนามดิจิทัล (ชื่อ, ตำแหน่ง, วันที่) และปิดผลประเมิน

### FR-6 การคำนวณคะแนนและแปลผล (Scoring & PLC Engine)
- FR-6.1 รวมคะแนน 5 องค์ประกอบเป็น /20 อัตโนมัติจากคะแนนที่ CAM ยืนยัน
- FR-6.2 แปลผลเป็น 1 ใน 4 ระดับตามช่วงคะแนน
- FR-6.3 แสดง PLC Action Trigger ตามระดับโดยอัตโนมัติ (รวมคำเตือน "ห้ามใช้สอนทันที" สำหรับ 9–12)

### FR-7 รายงานและประวัติ
- FR-7.1 สร้างรายงานผลประเมิน (PDF) ตรงรูปแบบต้นฉบับ: ส่วนที่ 1–4 + คะแนน + Feedback + ลายเซ็น
- FR-7.2 CAT ดาวน์โหลดรายงานและดู Feedback ได้เมื่อผลปิดแล้ว
- FR-7.3 เก็บประวัติการประเมินทุกเวอร์ชัน เพื่อเทียบพัฒนาการระหว่างรอบ

## 5. ความต้องการไม่ใช่ฟังก์ชัน (Non-Functional Requirements)

| รหัส | ด้าน | ข้อกำหนด |
|------|------|----------|
| NFR-1 | ภาษา | UI และการวิเคราะห์รองรับภาษาไทยเต็มรูปแบบ (รวม OCR ไทย) |
| NFR-2 | ประสิทธิภาพ | AI ประเมิน 1 แผน (≤30 หน้า) เสร็จภายใน ≤ 90 วินาที |
| NFR-3 | ความถูกต้อง | ทุกคะแนนที่ AI เสนอต้องมี Evidence Trail อ้างอิงได้จริง (ตรวจสอบย้อนกลับได้) |
| NFR-4 | ความเป็นส่วนตัว | ข้อมูลแผน/ครูเข้าถึงได้เฉพาะเจ้าของและ CAM ที่ดูแล; เข้ารหัสขณะจัดเก็บและส่ง |
| NFR-4.1 | ความปลอดภัย Auth | ใช้ Google OAuth 2.0/OIDC ผ่าน HTTPS เท่านั้น; ตรวจสอบ `state` (กัน CSRF) และ `nonce`; เก็บ Client Secret ใน env ฝั่งเซิร์ฟเวอร์ (ไม่หลุดฝั่ง client); session cookie ตั้ง `HttpOnly` + `Secure` + `SameSite` |
| NFR-5 | Auditability | เก็บ log ทุกการเปลี่ยนคะแนนและการยืนยัน พร้อม timestamp/ผู้กระทำ |
| NFR-6 | ความน่าเชื่อถือ AI | บันทึก prompt/รุ่นโมเดล/เวอร์ชัน rubric ที่ใช้ ต่อการประเมินแต่ละครั้ง (reproducibility) |
| NFR-7 | Usability | CAM ตรวจ 1 แผนจบใน ≤ 10 นาที ด้วยมุมมอง side-by-side |
| NFR-8 | ความพร้อมใช้ | รองรับใช้งานบนเบราว์เซอร์เดสก์ท็อปและแท็บเล็ต |
| NFR-9 | การ deploy (Base Path) | รันได้ภายใต้ sub-path ของโดเมน โดย `basePath` มาจาก env (ไม่ hardcode) — ครอบคลุมข้อกำหนดย่อย NFR-9.1 ถึง NFR-9.7 ด้านล่าง |

### NFR-9 รายละเอียดข้อกำหนด Base Path (Sub-folder Deployment)
- **NFR-9.1 Configurable base path** — `basePath` (เช่น `/assert-plan`) กำหนดผ่าน environment variable ครั้งเดียว เปลี่ยน sub-path ได้โดยไม่แก้โค้ด
- **NFR-9.2 Routing** — เส้นทางหน้า (page routes) ทั้งหมดทำงานถูกต้องเมื่ออยู่ใต้ `basePath`; ไม่มีลิงก์ที่ชี้กลับ root ของโดเมน
- **NFR-9.3 Static assets** — JS/CSS/รูป/**ฟอนต์ไทย** โหลดจาก `<basePath>/...` (asset prefix) ไม่ 404
- **NFR-9.4 API/Server routes** — endpoint (อัปโหลด, เรียก AI, บันทึกผล) อยู่ใต้ `<basePath>/api/...` และ client เรียกด้วย relative/prefixed path เสมอ
- **NFR-9.5 Cookies/Session** — cookie ตั้ง `Path=<basePath>` เพื่อ scope การล็อกอิน (CAT/CAM) ให้แยกจากแอปอื่นบนโดเมนเดียวกัน ป้องกันชนกัน
- **NFR-9.6 ลิงก์ในรายงาน/ไฟล์** — URL ดาวน์โหลดรายงาน PDF และลิงก์ไฟล์แผนที่อัปโหลด ต้องประกอบด้วย `basePath` (ทั้งลิงก์ในหน้าและลิงก์ที่ฝังในเอกสาร/อีเมลแจ้งเตือน ถ้ามี)
- **NFR-9.7 Reverse proxy compatibility** — ทำงานถูกต้องหลัง reverse proxy ที่เพิ่ม/ตัด path prefix; เคารพ header `X-Forwarded-Host`/`X-Forwarded-Proto` ในการสร้าง absolute URL (เช่น ลิงก์ในอีเมล)

## 6. หน้าจอหลัก (UI Screens — ระดับ SRS)
1. **หน้าเข้าสู่ระบบ** — แถวโลโก้พันธมิตร + ชื่อโครงการ + การ์ดฟีเจอร์ 3 ใบ + **ปุ่มเดียว "เข้าสู่ระบบด้วย Google" (Gmail)**; ไม่มีช่องรหัสผ่าน; แสดงข้อความปฏิเสธหากอีเมลนอกโดเมนที่อนุญาต (ดู UI จริงใน [UX/UI Design](UX_UI_Design_ALPR.md) §หน้า 1)
2. **CAT — อัปโหลดแผน** (ฟอร์มส่วนที่ 1 + drag-drop ไฟล์ + สถานะ)
3. **CAT — ผลประเมินของฉัน** (คะแนน /20, ระดับ, Feedback, ดาวน์โหลด PDF)
4. **CAM — คิวแผนรอตรวจ** (รายการ + สถานะ + confidence เฉลี่ยจาก AI)
5. **CAM — หน้าตรวจประเมิน** (⬅ ตัวแผน + Evidence | ➡ 5 องค์ประกอบให้ยืนยัน/แก้ + confidence)
6. **CAM — สรุปและลงนาม** (คะแนนรวม, ระดับ, PLC Action, ช่อง Strengths/Growth, ลายเซ็น)

## 8. เกณฑ์การยอมรับ (Acceptance Criteria)
- **AC-1** CAT อัปโหลด PDF สแกนภาษาไทยได้ และระบบสกัดข้อความผ่าน OCR สำเร็จ
- **AC-2** สำหรับแผนตัวอย่าง AI คืนคะแนน C1–C5 ครบ พร้อม Evidence Trail (ข้อความ+หน้า) ทุกองค์ประกอบ
- **AC-3** เมื่อไม่มีหลักฐาน AI คืน "ไม่พบหลักฐาน" + ระดับต่ำ (ไม่เดาคะแนนสูง)
- **AC-4** CAM แก้คะแนนได้ และระบบบันทึก audit (ค่า AI เดิม → ค่าใหม่)
- **AC-5** คะแนนรวม /20 และแถบระดับ + PLC Action คำนวณถูกต้องตามตารางในข้อ 3
- **AC-6** แผนที่ได้ 9–12 คะแนน แสดงคำเตือน "ห้ามนำไปใช้สอนทันที" เด่นชัด
- **AC-7** รายงาน PDF ออกตรงรูปแบบต้นฉบับ (ส่วนที่ 1–4) และ CAT ดาวน์โหลดได้หลังผลปิด
- **AC-8** ผลประเมินไม่ถือว่าสมบูรณ์จนกว่า CAM จะลงนามยืนยัน
- **AC-9** เมื่อตั้งค่า `basePath` (เช่น `/assert-plan`) แล้ว deploy: หน้าเว็บ, static asset (รวมฟอนต์ไทย), API, การล็อกอิน (cookie), และลิงก์ดาวน์โหลดรายงาน PDF ทำงานถูกต้องทั้งหมดใต้ `https://<domain>/<basePath>/` โดยไม่มี 404 หรือลิงก์หลุดไป root; เปลี่ยนค่า `basePath` แล้วใช้งานได้โดยไม่ต้องแก้โค้ด
- **AC-10** ผู้ใช้กด "เข้าสู่ระบบด้วย Google" แล้วล็อกอินสำเร็จด้วยบัญชี Gmail ในโดเมนที่อนุญาต และถูกนำเข้าสู่ workspace ตามบทบาท (CAT/CAM)
- **AC-11** อีเมล Google นอกโดเมนที่อนุญาต (allow-list) ถูกปฏิเสธพร้อมข้อความชัดเจน และไม่มีการสร้าง session
- **AC-12** callback ของ Google OAuth ทำงานถูกต้องเมื่ออยู่ใต้ `<basePath>` (redirect URI = `https://<domain>/<basePath>/api/auth/callback/google`) และ session cookie เป็น `HttpOnly`+`Secure`+`Path=<basePath>`

## 9. ประเด็นเปิด / ต้องยืนยันเพิ่ม (Open Questions)
1. ใช้โมเดล AI ตัวใด และ deploy ที่ใด (เช่น Claude ผ่าน API / on-prem เพื่อความเป็นส่วนตัวข้อมูลครู)?
2. ต้องรองรับ **หลายโรงเรียน/หลายเขต** ในระบบเดียวหรือไม่ (มีผลต่อ tenancy)?
   - **[อัปเดต] การ deploy:** ยืนยันแล้วว่าเป็น **sub-path ใต้โดเมนที่มีอยู่** (ดู NFR-9) — ยังต้องระบุ: ชื่อ `basePath` จริง, โดเมนปลายทาง, และเป็น reverse-proxy แบบใด (Nginx/Vercel rewrites/อื่น ๆ)?
3. ระดับความเป็นทางการของ **ลายเซ็นดิจิทัล** (ภาพลายเซ็น vs. e-signature ตามกฎหมาย)?
4. ต้องการ export ข้อมูลดิบ (คะแนน+audit) เพื่อการวิเคราะห์วิจัยในรูป CSV/Sheet หรือไม่?
5. ในอนาคตจะผนวกฟอร์มฉบับ 1 และ 2 เข้าระบบเดียวกันหรือแยกระบบ?
